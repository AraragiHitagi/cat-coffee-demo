# AI 第一课学习总结

日期：2026-04-08

## 主题

这次学习主要围绕两个问题展开：

1. 如何通过命令行以最小方式启动不同的 Agent CLI
2. 如何理解“元规则”在真实工程中的作用和落地方式

## 一、CLI 启动的核心思路

如果官方没有成熟的 Agent SDK，不必卡死在 SDK 路线里。只要有 CLI，就可以把 CLI 当作 agent runtime 来用。

最小实现通常只有几步：

1. 用 `child_process.spawn()` 启动 CLI
2. 用 `readline` 按行读取 stdout
3. 对每一行做 JSON 解析
4. 提取真正的回复文本
5. 处理 `stderr` 和退出码

这个思路已经分别在：

- `minimal-codex.js`
- `minimal-opencode.js`

中实现。

## 二、为什么还要做统一 invoke 封装

虽然 `codex` 和 `opencode` 都能被命令行调起，但它们并不共享同一套协议：

- 启动参数不同
- JSON 事件格式不同
- 文本字段位置不同
- stderr 噪音特征也不同

所以抽象层不应该假装它们完全一样，而应该做一个“统一调用接口 + 分 provider 适配”的结构。

这就是 `invoke.js` 的作用：

- 对外统一成 `invoke(cli, prompt)`
- 对内根据 `cli` 选择参数、解析器和 stderr 过滤策略

这个抽象更稳，因为它承认差异，而不是掩盖差异。

## 三、元规则真正解决什么问题

元规则不是为了让 AI “更听话”，而是为了修正 AI 在工程协作中的几个天然弱点：

- 容易幻觉
- 容易讨好
- 容易过度自信
- 容易忘记上下文

所以好的元规则，本质上是在给 AI 打补丁。

例如：

- 要求交接时必须写 `Why`
- 要求 review 不能先表演性认同
- 要求声称完成之前必须有验证证据
- 要求 merge 前必须经过 reviewer 放行

## 四、clowder-ai 给出的启发

在 `clowder-ai` 里，元规则不是一段抽象说教，而是拆成了四层：

1. 入口规则文件
2. 共享规则文件
3. 场景化 Skills
4. SOP 工作流

这带来的启发是：

- prompt 只是起点，不是终点
- 真正可复用的经验应该写成规则系统
- 真正可靠的协作应该有硬门禁，而不是靠模型“自觉”

## 五、当前产出

这次学习最终沉淀了几个直接可用的结果：

- 一个 `codex` 最小启动脚本
- 一个 `opencode` 最小启动脚本
- 一个统一的 `invoke(cli, prompt)` 调用层
- 一份关于 `clowder-ai` 元规则落地方式的笔记

## 六、一句话结论

第一课最重要的收获是：

当 SDK 不够用时，可以从 CLI 接管 agent；当 prompt 不够稳时，要把经验升级成规则系统。
