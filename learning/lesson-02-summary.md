# AI 第二课学习总结

日期：2026-04-08

## 主题

第二课的核心不是“怎么把 CLI 跑起来”，而是：

如何让一个已经能跑的 CLI 方案，不至于在真实协作和真实数据面前连续翻车。

作者在这一课里讲的不是抽象最佳实践，而是一串真实事故：

- 幻觉导致错误诊断
- 只监听 stdout 导致误杀进程
- Redis 重启导致关键记录丢失
- 开发环境和用户环境没有充分隔离

## 一、最重要的教训：stderr 也是活跃信号

这是第二课最关键、最容易踩的坑。

错误做法：

- 只监听 `stdout`
- 认为“几分钟没有 stdout 输出”就说明 CLI 卡死
- 然后直接强制 kill 进程

问题在于，很多 Agent CLI 在这些阶段并不往 `stdout` 输出最终文本：

- thinking
- 工具调用
- 中间进度

这些内容往往会进 `stderr`。

所以如果只看 `stdout`，就会把“正在认真工作”的 agent 误判成“死掉了”。

正确做法：

- `stdout` 和 `stderr` 都要视为活跃信号
- 超时逻辑要基于“最近活跃时间”而不是“最近最终回答时间”

一句话说就是：

不要把 `stderr` 当成“纯错误流”，在 CLI agent 场景里，它经常也是状态流。

## 二、第二个教训：AI 会在信息不足时自信瞎猜

课里那个“狼人杀暗号测试”的本质，不是一个趣味故事，而是在提醒：

- AI 会把自己已知的信息误当成别人说过的话
- 信息不足时，它不一定会说“我不知道”
- 它更可能拼出一个格式完整、语气自信、但事实错误的答案

这说明对 AI 的验证，不能只看：

- 有没有回答
- 回答得像不像样

还要看：

- 这个答案的证据链在哪
- 它到底是“看到了”还是“猜到了”

所以工程上应该补一条非常实用的规则：

- 不确定时要允许 AI 停下来提问
- 不要奖励它“先给一个像样答案再说”

## 三、第三个教训：开发环境必须隔离

第二课里另一个非常硬核的教训是 Redis 数据丢失。

问题不是 Redis 本身不好，而是：

- 测试和开发动作碰到了有价值的数据
- 重启、快照、持久化周期这些基础设施行为，没有被当成真实风险

作者后来明确加了一个隔离原则：

- 用户数据一套 Redis
- 开发测试一套 Redis
- Worktree 开发必须连开发 Redis，不准碰用户 Redis

这个启发很重要：

当你开始让 AI 帮你改代码、重启服务、跑迁移时，数据库隔离已经不是“可选优化”，而是底线。

## 四、第四个教训：备份不是可有可无

冠军记录之所以还能被恢复认定，不是因为系统设计得多完美，而是因为铲屎官提前做了备份。

这说明：

- 任何一次“看起来只是重启一下”的操作，都可能造成不可逆损失
- 真正的最后一道防线，不是 AI 的聪明，而是备份

所以从工程角度看，第二课其实在强调一个现实主义原则：

不要假设系统永远不出错，要假设出错之后你还能不能恢复。

## 五、这课真正补上的是什么

第一课解决的是：

- 你能不能调起 CLI

第二课解决的是：

- 你有没有把 CLI 当成真实生产部件来对待

这中间最大的差别在于是否具备这些意识：

- 进程活跃信号不能只看 stdout
- 超时策略不能拍脑袋
- AI 的回答要防幻觉
- 数据存储必须隔离
- 重要操作前要有备份

## 六、我从第二课学到的工程 checklist

以后只要我写 CLI agent 封装，至少应该检查这几件事：

1. 是否同时监听了 `stdout` 和 `stderr`
2. 是否把 `stderr` 视为活跃信号，而不是一概当噪音
3. 超时逻辑是否会误杀长思考、长工具调用任务
4. 是否允许 agent 在不确定时停下来提问
5. 开发环境是否和真实用户数据完全隔离
6. 高风险操作前是否有备份或回滚方案

## 七、一句话结论

第二课最重要的结论是：

从“能跑”到“能用”的距离，往往就藏在 `stderr`、超时、幻觉、隔离和备份这些看起来不性感、但一踩就炸的工程细节里。
